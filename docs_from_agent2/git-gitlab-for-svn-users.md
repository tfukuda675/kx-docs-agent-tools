# SVN利用者向け Git / GitLab よく使う機能と利用シーン

## Gitのよく使う機能

| 機能 | どんな状況で使うか |
|---|---|
| `clone` | GitLab上のリポジトリを自分のPCに持ってきて作業を始めるとき |
| `status` | 今、自分の作業ディレクトリで何が変更されているか確認するとき |
| `diff` | 変更内容をコミット前に確認するとき |
| `add` | コミットに含めたい変更を選ぶとき |
| `commit` | 作業内容をローカル履歴として記録するとき |
| `branch` | バグ修正、機能追加、実験などを本流から分けて作業するとき |
| `checkout` / `switch` | 作業するブランチを切り替えるとき |
| `merge` | あるブランチの変更を別のブランチに取り込むとき |
| `rebase` | 自分の作業ブランチを最新の本流に載せ直したいとき |
| `pull` | GitLab上の最新変更を自分の環境に取り込むとき |
| `push` | 自分のローカルコミットをGitLabに送って共有するとき |
| `fetch` | GitLab上の最新情報だけ取得し、作業ツリーにはまだ反映しないとき |
| `log` | 過去のコミット履歴を確認するとき |
| `blame` | ある行を誰がいつ変更したか調べるとき |
| `stash` | 作業途中の変更を一時退避して、別作業に切り替えるとき |
| `reset` | ローカルの変更やコミットを巻き戻したいとき |
| `revert` | 共有済みのコミットを安全に打ち消したいとき |
| `tag` | リリース版など、特定の時点に名前を付けたいとき |
| `cherry-pick` | 特定のコミットだけを別ブランチに取り込みたいとき |

## GitLabのよく使う機能

| 機能 | どんな状況で使うか |
|---|---|
| Repository | GitリポジトリをWeb上で確認・管理するとき |
| Issue | バグ、要望、作業タスクを記録・管理するとき |
| Merge Request | 自分の変更を本流に取り込んでもらうため、レビューやCI確認を依頼するとき |
| Code Review | 変更内容に対してコメントや修正依頼をするとき |
| CI/CD Pipeline | pushやMerge Requestをきっかけに、自動テストやビルドを実行するとき |
| Branch Protection | `main` や `develop` に直接pushできないようにして、レビュー必須にするとき |
| Labels | IssueやMerge Requestを `bug`、`urgent`、`frontend` などで分類するとき |
| Milestone | リリースやスプリント単位でIssueやMerge Requestをまとめるとき |
| Assignee | IssueやMerge Requestの担当者を明確にするとき |
| Reviewers / Approvers | 誰にレビュー・承認してもらうか指定するとき |
| Comments / Threads | 仕様相談、修正依頼、確認事項を履歴として残すとき |
| Wiki / Pages | 開発手順や設計メモなどをチームで共有するとき |
| Releases | リリース内容、タグ、成果物をGitLab上で管理するとき |

## 機能ごとの使い方イメージ

### `branch`

`branch` は、作業を分離するために使います。

たとえば「バグ修正」「新機能追加」「設計変更の試作」を、安定している `main` や `develop` から切り離して進めたいときに使います。

SVNのように全員が同じ場所を直接触る感覚ではなく、「自分の作業場所を作る」イメージです。

### `merge`

`merge` は、別ブランチの変更を取り込むときに使います。

たとえば `feature/login` で作ったログイン機能を `develop` に取り込む場合です。

GitLabでは通常、直接mergeするのではなく、Merge Requestを経由してレビューやCI確認をしてから取り込みます。

### `rebase`

`rebase` は、自分の作業ブランチの土台を最新化するときに使います。

たとえば、自分が作業している間に `develop` が進んでしまった場合、自分の変更を最新の `develop` の上に載せ直します。

履歴をきれいに保てますが、共有済みブランチで不用意に使うと混乱しやすいため、チームルールが必要です。

### `reset`

`reset` は、ローカル作業を巻き戻すときに使います。

「直前のコミットをやり直したい」「まだpushしていない作業を取り消したい」場合に便利です。

ただし、共有済み履歴に対して使うと危険なので、基本はローカル限定の操作として説明するのが安全です。

### `revert`

`revert` は、過去の変更を打ち消す新しいコミットを作るときに使います。

すでにGitLabへpush済み、または他人が取り込んでいるコミットを取り消す場合はこちらを使います。

履歴を壊さず「取り消した」という記録も残るため、チーム開発では `reset` より安全です。

### `checkout` / `switch`

`checkout` / `switch` は、作業対象を切り替えるときに使います。

「今はバグ修正ブランチで作業する」「次はレビュー指摘対応ブランチに戻る」といった場面です。

また、過去のコミット時点を一時的に確認する場合にも使います。

### `stash`

`stash` は、作業途中の変更を一時的に避難させるときに使います。

たとえば、まだコミットできない途中状態だけれど、急ぎのバグ修正のために別ブランチへ移動したい場合です。

### `cherry-pick`

`cherry-pick` は、特定のコミットだけを別ブランチに持っていくときに使います。

たとえば、`develop` に入れたバグ修正だけを、リリースブランチにも取り込みたい場合です。

### `tag`

`tag` は、特定時点に名前を付けるときに使います。

たとえば `v1.0.0` や `release-2026-06-14` のように、リリースしたコミットを後から特定できるようにします。

### `fetch` / `pull`

`fetch` と `pull` は、GitLab上の変更を取得するときに使います。

`fetch` は「情報だけ取ってくる」、`pull` は「取ってきて今のブランチに反映する」です。

SVNの `update` に近いのは `pull` ですが、Gitではブランチやmergeの考え方が絡むため、完全に同じではありません。

### `push`

`push` は、自分のコミットをGitLabに送るときに使います。

Gitでは `commit` しただけでは他人に見えません。

チームに共有するには `push` が必要です。

### Merge Request

Merge Request は、GitLabで最も重要なチーム開発機能の一つです。

自分のブランチの変更を `develop` や `main` に取り込んでもらうための依頼です。

レビュー、コメント、CI結果、承認、mergeまでを一つの画面で扱えます。

### Issue

Issue は、作業や問題を管理するために使います。

「バグがある」「仕様を確認したい」「この機能を追加したい」といった内容を記録し、担当者、ラベル、期限、関連Merge Requestを紐づけます。

### CI/CD Pipeline

CI/CD Pipeline は、自動確認のために使います。

pushやMerge Request作成時に、自動でテスト、ビルド、静的解析などを実行します。

「人が見る前に機械的に壊れていないか確認する」仕組みです。


## よく使うコマンドオプション

ここでは、選択肢が多くて迷いやすいコマンドを中心に、チーム開発でよく使うオプションを説明します。

### `merge` のよく使うオプション

| コマンド例 | どんな状況で使うか |
|---|---|
| `git merge ブランチ名` | 指定したブランチの変更を、今いるブランチに取り込むとき |
| `git merge --no-ff ブランチ名` | fast-forwardできる場合でもmergeコミットを作り、ブランチ単位の作業履歴を残したいとき |
| `git merge --ff-only ブランチ名` | 履歴が分岐していない場合だけ取り込み、mergeコミットを作りたくないとき |
| `git merge --squash ブランチ名` | 作業ブランチ上の複数コミットを1つの変更としてまとめて取り込みたいとき |
| `git merge --abort` | merge中に競合が発生し、いったんmerge開始前の状態へ戻したいとき |

`--no-ff` は「この機能はこのブランチで作った」という履歴を残したい場合に向いています。

`--ff-only` は、意図しないmergeコミットを作らないための確認として使えます。自動化やCIで安全に取り込めるか確認したい場合にも使われます。

`--squash` は履歴をすっきりさせたい場合に便利ですが、元の細かいコミット履歴は取り込み先には残りません。

### `rebase` のよく使うオプション

| コマンド例 | どんな状況で使うか |
|---|---|
| `git rebase ブランチ名` | 自分の作業ブランチを、指定したブランチの最新状態の上に載せ直したいとき |
| `git rebase -i ブランチ名` | コミットをまとめる、順番を変える、メッセージを直すなど、履歴を整理したいとき |
| `git rebase --continue` | rebase中の競合を解決したあと、rebase処理を続けるとき |
| `git rebase --abort` | rebase中に問題が起き、rebase開始前の状態へ戻したいとき |
| `git rebase --skip` | rebase中に適用できないコミットを飛ばして続行したいとき |
| `git rebase --autostash ブランチ名` | 未コミット変更がある状態で、一時退避してからrebaseし、最後に戻したいとき |

`-i` は便利ですが、履歴を書き換える操作です。基本的には、まだ他人が使っていない自分の作業ブランチで使います。

`--abort` は、競合解消が難しい場合や、rebaseするブランチを間違えた場合の戻り道として重要です。

`--skip` は慎重に使います。飛ばしたコミットの変更は取り込まれないため、本当に不要な変更か確認してから使います。

### `checkout` / `switch` のよく使うオプション

| コマンド例 | どんな状況で使うか |
|---|---|
| `git switch ブランチ名` | 既存ブランチに切り替えるとき |
| `git switch -c 新ブランチ名` | 今いる地点から新しいブランチを作り、そのブランチに切り替えるとき |
| `git switch -c 新ブランチ名 起点` | 過去のコミット、タグ、別ブランチなど、指定した時点から新しいブランチを切るとき |
| `git switch -` | 直前にいたブランチへ戻るとき |
| `git checkout コミットID` | 過去の時点の内容を一時的に確認したいとき |
| `git checkout -- ファイル名` | ファイル単位で作業ツリーの変更を取り消したいとき |

現在は、ブランチ切り替えには `switch` を使う方が意図が明確です。

`checkout` は古くからあるコマンドで、ブランチ切り替え、過去時点の確認、ファイル復元など複数の役割を持ちます。そのため、説明では「ブランチ切り替えは `switch`」「ファイル復元は `restore`」と分けると理解しやすくなります。

`git checkout コミットID` は detached HEAD という状態になります。過去の内容を見るだけなら問題ありませんが、その状態で作業を続ける場合は `git switch -c 新ブランチ名` でブランチを作ります。

### `reset` のよく使うオプション

| コマンド例 | どんな状況で使うか |
|---|---|
| `git reset --soft HEAD~1` | 直前のコミットだけ取り消し、変更内容はステージ済みのまま残したいとき |
| `git reset --mixed HEAD~1` | 直前のコミットだけ取り消し、変更内容は作業ツリーに残したいとき |
| `git reset HEAD ファイル名` | `git add` したファイルをステージから外したいとき |
| `git reset --hard HEAD~1` | 直前のコミットと作業ツリーの変更をまとめて捨てたいとき |
| `git reset --hard origin/ブランチ名` | ローカルブランチをリモートブランチと同じ状態に戻したいとき |

`--soft` は「コミットメッセージやコミットの分け方をやり直したいが、変更内容はそのまま残したい」ときに使います。

`--mixed` は `reset` の標準動作です。コミットは取り消すが、ファイル変更は残したい場合に使います。

`--hard` は作業ツリーの変更も消します。戻せない変更が出やすいため、未pushの自分の作業に限定して慎重に使います。共有済みのコミットを取り消す場合は、通常 `revert` を使います。

### 不足を補う関連コマンドとオプション

`checkout` や `reset` は役割が広いため、最近のGitでは目的別に `restore` や `switch` を使うと説明しやすくなります。また、チーム開発では `cherry-pick` や `stash` のオプションもよく使います。

#### `restore`

| コマンド例 | どんな状況で使うか |
|---|---|
| `git restore ファイル名` | 作業ツリー上のファイル変更を取り消したいとき |
| `git restore --staged ファイル名` | `git add` したファイルをステージから外したいとき |
| `git restore --source コミットID ファイル名` | 特定の過去時点から、指定ファイルだけを戻したいとき |

ファイル単位の取り消しや復元は、`checkout` や `reset` より `restore` を使う方が目的を説明しやすいです。

#### `cherry-pick`

| コマンド例 | どんな状況で使うか |
|---|---|
| `git cherry-pick コミットID` | 特定のコミットだけを今のブランチへ取り込みたいとき |
| `git cherry-pick -n コミットID` | コミットは作らず、変更内容だけを作業ツリーに取り込みたいとき |
| `git cherry-pick --continue` | cherry-pick中の競合を解決したあと、処理を続けるとき |
| `git cherry-pick --abort` | cherry-pick中に問題が起き、開始前の状態へ戻したいとき |

リリースブランチへバグ修正だけを持っていく場合などに使います。似た変更を複数ブランチへ入れる場合は便利ですが、同じ変更が別コミットとして複製される点には注意します。

#### `stash`

| コマンド例 | どんな状況で使うか |
|---|---|
| `git stash push -m "説明"` | 作業途中の変更に説明を付けて一時退避したいとき |
| `git stash push -u -m "説明"` | 未追跡ファイルも含めて一時退避したいとき |
| `git stash list` | 退避した作業の一覧を確認したいとき |
| `git stash pop` | 退避した作業を戻し、stash一覧から削除したいとき |
| `git stash apply` | 退避した作業を戻すが、stash一覧には残したいとき |

急ぎの割り込み作業でブランチを切り替えたいが、今の変更はまだコミットしたくない場合に使います。

## SVN利用者向けの対比

| SVNでの感覚 | Git / GitLabでの考え方 |
|---|---|
| updateして作業する | pull / fetchして最新状態を確認する |
| commitすると共有される | commitはローカル記録、pushして初めて共有 |
| trunkに直接コミット | branchで作業し、Merge Requestで取り込む |
| 履歴は中央にある | 各自のローカルにも履歴がある |
| 差分確認してコミット | addでコミット対象を選び、commitする |
| 競合はupdate / commit時に出る | merge / rebase / pull時に競合することが多い |
